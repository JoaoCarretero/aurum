  shim via importlib mas isso também é insuficiente — Python escolhe o
  ficheiro `.py` sobre o dir homônimo, tornando o shim dead-code.

## Achados

1. **Python import resolution — colisão file vs package:** quando existem
   `core/X.py` e `core/X/__init__.py` ao mesmo tempo, Python 3.14 escolhe
   o **ficheiro** `.py`. O `__init__.py` do pacote homônimo vira dead-code.
   Descoberto ao investigar comportamento do shim criado em Phase 3A para
   `core/signals/`.

2. **Runtime monkey-patch em `core/data/htf.py`:** função `prepare_htf`
   pega `sys.modules["core.signals"]` e atribui `CHOP_S21`, `CHOP_S200`,
   `MIN_STOP_PCT` para aplicar params HTF-específicos antes de chamar
   `indicators()`, `swing_structure()`, `omega()`. Requer que o módulo
   `core.signals` seja EXATAMENTE o módulo cujo código lê essas constantes.
   Subpacote `__init__.py` que re-exporta via `from ... import *` cria
   namespace separado e quebra o contrato.

3. **Path anchors em 40+ scripts de `tools/`** — padrão `Path(__file__).parent.parent`
   usado para derivar REPO root. Após mover para subdir, precisa de mais um
   `.parent`. Corrigido nos 40 scripts movidos. Mesmo padrão em 4 ficheiros
   de `core/` (alchemy_ui, analysis_export, engine_picker, fixture_capture).

4. **`data/` é gitignored, mas `core/data/` também match** — precisei de
   exceção explícita no `.gitignore`: `!core/data/` + `!core/data/**` com
   re-exclusão de pycache.

5. **`tests/tests/` orphan dir** — artefato pré-existente (não tracked) de
   algum teste phase_c que usava ROOT errado. Não afeta nada, mas fica
   registrado para limpeza manual.
