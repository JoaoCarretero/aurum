## Mudanças Críticas
**ATENÇÃO** — alteração em `engines/millennium.py::_collect_live_signals`:

Antes: 1 fetch shared (INTERVAL=15m + SYMBOLS=default/11 altcoins) consumido pelos 3 engines do pod. Signals divergiam do backtest pra JUMP (native 1h bluechip) e RENAISSANCE (native 15m bluechip).

Depois: cada engine faz fetch dedicado no seu sweet-spot via `_scan_one_engine_live(engine_name)`, novo helper em millennium.py que:
1. Lê `ENGINE_INTERVALS[engine]` + `ENGINE_BASKETS[engine]` de `config/params.py`
2. Chama `fetch_all(symbols, interval=tf, n_candles=SCAN_DAYS×bars_per_day)`
3. Computa `detect_macro` + `build_corr_matrix` no set dedicado
4. Chama a scan fn nativa (`scan_symbol` / `scan_mercurio` / `scan_hermes`) com `live_mode=True`
5. Stamp `strategy=ENGINE_UPPER`

Impacto na lógica de trading: **signals pod-MILLENNIUM agora batem com signals dos engines solo** (bit-identical pra mesmo candle). Antes, JUMP dentro do pod era diferente de JUMP solo (e ambos diferentes do backtest validated). Agora todos convergem.

Backtest path (`_collect_operational_trades` → `_scan_azoth` + `_scan_hermes_all`) **não tocado** — MILLENNIUM backtest 360d Sharpe 8.22 ROI 87% DD 2.78% continua igual.

Performance: MILLENNIUM tick aumentou de ~0.7s → ~2s (3 fetches em vez de 1). Irrelevante pro tick_sec=900.
