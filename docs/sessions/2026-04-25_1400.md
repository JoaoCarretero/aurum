## Achados

- **TTL cache reduces FS load 6x**: scan_artifacts estava sendo chamado a cada 5s (poll) walking todo `docs/`, `data/`, e git refs. Em devs ativos com hundreds de runs em data/ + 40+ session/daily logs, isso adicionava ~50-100ms de blocking no main thread por tick. TTL=30s mantem a mesma logica funcional mas reduz o fan-out 6x. Issues continuam fluindo no merge_events a cada tick (issues sao parameter, nao cached).

- **Stale snapshot agora surface error**: antes, scanner falhava → fallback pra `_last_full_scan`, sem indicador. Permanente fail = UI mostrava data de horas atras como se fosse fresca. Agora apos 2x TTL (60s), flash + log.

- **Cumulative sparkline foi cosmetic broken**: `monthly_spent_cents` do Paperclip e cumulative — plot de 30d mostrava monotone-ascending line resetando dia 1. Inutil como sparkline. Agora plota daily burn — visivel "spike no dia X" significa user vendo onde gastou, nao quanto acumulou.

- **stats_db check_same_thread=False**: nao habilita threading hoje (so main thread acessa), mas remove o sys-level error caso alguem mova cost_dashboard fetcher pra background thread no futuro. SQLite WAL ja garante db-level concurrency.
