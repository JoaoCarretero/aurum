## Achados

### Achado 1 — 6 camadas de bug causavam flicker + travamento no Live Cockpit

Em ordem de severidade:

- **Detail pane re-rendering a cada refresh** (mesmo sem mudança real)
  → fix: sig-check no `_render_detail` (commit `5ec4979`).
- **Master-list rebuild a cada worker completion** → fix: sig-skip no
  `_render_master_list` (commit `e7c2e07`).
- **Refresh stampede via `after(1ms)`** quando 3 workers completavam
  em paralelo → fix: coalesce flag + 150ms debounce (commit `5a9753a`).
- **TTLs 2s forçando rebuild a cada 8s** → fix: TTL 5s → 60s (commit
  `9fda35b`).
- **Sig compute trancava no lock** do `_COCKPIT_RUNS_CACHE` enquanto
  worker HTTP rodava (até 5s timeout) → fix: `_paper_content_sig_cheap`
  e `_shadow_content_sig_cheap` lêem só cache via dict.get atomic,
  nunca bloqueiam (commit `79f2928`).
- **`collect_vps_runs` 6s por chamada**: 48 HTTP calls (/heartbeat +
  /account por run * 24 runs) através do SSH tunnel → fix: `/v1/runs`
  agora serve payload completo (ticks_ok, equity, etc), `collect_vps_runs`
  é single round-trip (commit `1bb30a4`).

### Achado 2 — Warmup thread disparava antes do SSH tunnel subir

Launcher boot: warmup era dispatched 25ms depois de `_start_tunnel_async`
ser agendado, mas o tunnel leva 1-3s para subir. `_load_cockpit_runs_sync`
batia em `127.0.0.1:8787` antes do tunnel aceitar → connection refused
→ cache preenchido com `[]` → próximos 60s (TTL) o cockpit mostrava
"nenhum run" mesmo com VPS rodando 4 runs.

Fix: warmup thread agora polla tunnel.status==UP por até 15s antes
de fazer o fetch HTTP. Se tunnel nunca sobe, bail out; próxima
interação do user dispara novo fetch assincronamente (commit `21252ea`).

### Achado 3 — Multi-instance picker existia mas era invisível

Picker rodava com styling font 7 + cor DIM no BG do PANEL, quase
invisível ao lado das outras panels densas. Reconstruído como
"RUNNING NOW" strip institucional: amber rule + caps heading, vertical
stacked rows (escalável pra N runs com scroll acima de 6), dot
verde/cinza + mode badge colorido + label + tick count. Filtrado
por mode atual (paper ou shadow separadamente) — cross-mode ficou
confuso (commit `1adb785` depois `00a03d5` depois `d554f76`).

### Achado 4 — Counter "RUNNING" mostrava engines únicos, não instâncias

`_vps_running_slugs` retornava set deduplicado. MILLENNIUM com 2
paper + 2 shadow = 1 slug. Novo `_vps_running_instance_count` conta
linhas reais do cache filtradas por mode (commit `930696d`).

### Achado 5 — DB stale "running" entries vazando pro picker

DB local tinha entries antigas `status=running` do dia anterior
(bug do paper graceful stop que não marca stopped no heartbeat).
Picker mostrava VPS real + ghost DB mixed. Fix: quando VPS cache
tem runs do mode, usar VPS como source of truth e ignorar DB rows
que não batem com VPS run_id (commit `5e5e005`).

### Achado 6 — Close do launcher hangava

`_quit()` + `_shutdown_runtime()` chamavam `tunnel.stop(timeout_sec=3s)`
em duplicata (até 6s wait). Warmup thread com HTTP sync pendente
também segurava o processo. Fixes: timeouts 3s→1s / 1s→0.5s, remoção
da stop duplicada, warmup thread checa `app._ui_alive` antes de cada
poll (commit `8ea5c9c`).
