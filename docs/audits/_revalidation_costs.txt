# Cost Symmetry Audit — 2026-04-17

## Canonical cost path (core.signals.label_trade)

label_trade() is a PATH simulator only — it returns (result, duration, exit_price,
exit_reason). It does NOT apply costs directly. Costs are computed by the
caller immediately after label_trade returns.

The entry-side cost (SLIPPAGE+SPREAD) is baked into the entry price before
label_trade is called, via calc_levels() in core/signals.py:

  core/signals.py:211  slip = SLIPPAGE + SPREAD
  core/signals.py:212  entry = raw * (1 + slip)   # BULLISH
  core/signals.py:219  entry = raw * (1 - slip)   # BEARISH

The exit-side costs (COMMISSION x2, SLIPPAGE+SPREAD) are applied post-label in
each engine's inline PnL block. FUNDING is derived from duration returned by
label_trade.

Summary of what each engine must do for full C1+C2 coverage:
  Entry side: SLIPPAGE+SPREAD baked into entry price (via calc_levels or equivalent)
  PnL block:  COMMISSION on entry_cost + (COMMISSION + SLIPPAGE + SPREAD) on exit + FUNDING

Components coverage in label_trade itself:
- SLIPPAGE:   ✗ (applied upstream in calc_levels, not inside label_trade)
- SPREAD:     ✗ (applied upstream in calc_levels, not inside label_trade)
- COMMISSION: ✗ (applied in engine PnL block after label_trade)
- FUNDING:    ✗ (applied in engine PnL block using duration from label_trade)

Note: this is by design. label_trade is a path simulator, not a cost applier.

## Bug-suspect flags

### FLAG 1 — KEPOS: entry SLIPPAGE+SPREAD missing

  engines/kepos.py:260
      entry = float(df["open"].iloc[t + 1])
      # No slip applied. Entry is raw next-bar open.

  engines/kepos.py:496-506 (PnL function _pnl_with_costs)
      slip_exit = SLIPPAGE + SPREAD   # ← applied only to EXIT side
      entry_cost = entry * (1 + COMMISSION)   # ← only COMMISSION on entry

  Impact: entry side understated by SLIPPAGE+SPREAD = 0.03% per trade.
  For a long trade: entry_cost should be entry*(1+0.0003) but is entry*(1+0).
  Net effect: PnL overstated by ~0.03% of entry notional per trade on entry.
  Combined with symmetric overstatement on the exit side being correctly charged,
  the bias is entry-side only: PnL inflated by (SLIPPAGE+SPREAD)*entry*size*LEVERAGE.

  Magnitude estimate: 0.03% × avg_entry × avg_size × LEVERAGE per trade.
  At LEVERAGE=10, avg_entry=100, avg_size=1: ~$0.30/trade.
  Depends on trade count and position size. For ~500 trades: ~$150 total bias.

### FLAG 2 — MEDALLION: entry SLIPPAGE+SPREAD missing (same pattern as KEPOS)

  engines/medallion.py:438
      entry = float(df["open"].iloc[t + 1])
      # No slip applied. Entry is raw next-bar open.

  engines/medallion.py:660-675 (PnL function _pnl_with_costs)
      Comment reads: "Identical to citadel/kepos — intentional"
      # This comment is misleading: CITADEL applies slip via calc_levels
      # KEPOS/MEDALLION do not. The PnL function IS identical, but the
      # input `entry` is NOT equivalent.

  Impact: same as KEPOS. PnL inflated by ~SLIPPAGE+SPREAD per trade on entry.

  Note: KEPOS and MEDALLION both failed OOS on 2026-04-16, so this bug
  overstated their in-sample performance. It does not change their OOS verdict
  (already flagged red), but confirms the in-sample Sharpe was inflated.


## Notes on Bridgewater (priority suspect per task brief)

BRIDGEWATER (claim: OOS Sharpe 11.04 / ROI 267%) was the primary concern.

Static analysis shows cost coverage is COMPLETE for BRIDGEWATER:
- Entry: uses calc_levels (core/signals.py:211) → SLIPPAGE+SPREAD baked in

- No backtest_mode or cost-zeroing flag found anywhere

The anomalous OOS Sharpe cannot be explained by missing costs. Other
explanations to investigate (not in scope of this audit):

  - Sentiment data (funding rates, OI, LS) using future values
  See docs/audits/2026-04-16_oos_verdict.md for BRIDGEWATER bug-suspect classification.
