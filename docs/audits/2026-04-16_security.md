# Security Audit — 2026-04-16

## Sumário executivo

O AURUM Finance apresenta uma postura de segurança **acima da média** para um sistema quant de uso pessoal, com arquitectura defensiva madura nas camadas críticas: risk gates multi-camada em `core/risk_gates.py`, kill-switch explícito com ordem correcta (flatten antes de cancel) em `engines/live.py`, audit trail append-only com hash-chain opcional em `core/audit_trail.py`, e verificação de identidade de processos no `core/proc.py` (defesa contra PID recycling no Windows). Secrets reais nunca entraram no histórico do git.

**Top 3 riscos identificados:**
1. **Secrets em plaintext no disco** (`config/keys.json`) com credenciais demo Binance, bot token Telegram, FRED e NewsAPI — OneDrive sync e backups incontrolados expandem a superfície de ataque massivamente.
2. **API NEXUS bind em `0.0.0.0`** (`run_api.py:13`) expondo endpoints admin de `/api/trading/start|stop` e WebSocket de controlo de engines a qualquer máquina na rede local/LAN/VPN.
3. **`pickle.load` sobre ficheiros em `data/.cache/`** (`core/cache.py:50,69`) — deserialização insegura que converte qualquer escrita ao directório de cache numa RCE silenciosa dentro do processo da engine.

Nível geral: **MEDIUM** (com as 3 correcções críticas endereçadas, cai para LOW).

## Findings

### Critical

#### C1. Secrets reais em plaintext no disco dentro de pasta OneDrive
**Ficheiro:** `config/keys.json` (linhas 4-25)

Contém, como plaintext no disco:
- Chaves demo Binance (`api_key`, `api_secret`) — expõem a conta demo mas não mainnet.
- **Bot token Telegram + chat_id** — quem roubar este ficheiro controla completamente o bot AURUM e tem acesso à persona do operador.
- **FRED API key + NewsAPI key** — tokens pessoais do operador, uso quotável ao Joao.

O `config/keys.json` **está gitignored** (`.gitignore:22`) e **nunca foi committed** (verificado via `git log --all --full-history` — sem hits, e `git log -S` com valores específicos — sem hits). Porém:

1. O projecto vive em `C:\Users\Joao\OneDrive\aurum.finance\` — **OneDrive sincroniza com a cloud Microsoft**, replica em outros dispositivos Joao, e qualquer share accidental do directório pai expõe as chaves.
2. O módulo `core/key_store.py` existe e implementa Fernet + PBKDF2 (480k iterações SHA-256) com lock/unlock em memória, e `tools/encrypt_keys.py` faz a migração. Mas **o ficheiro encrypted (`keys.json.enc`) NÃO existe** no filesystem, i.e. a migração nunca foi executada.
3. Para modo LIVE, `engines/live.py:273-283` emite apenas um WARNING quando usa plaintext, não recusa o arranque.

**Recomendação:** Executar `python tools/encrypt_keys.py`, definir `AURUM_KEY_PASSWORD` na env do launcher, confirmar o log "Keys loaded from ENCRYPTED store", **apagar `config/keys.json` manualmente**, e para LIVE considerar tornar a leitura de plaintext um erro fatal em vez de warning. Rotacionar todas as keys presentes no ficheiro actual (demo Binance, Telegram bot, FRED, NewsAPI) porque devem ser consideradas comprometidas dado o tempo que viveram em plaintext + OneDrive.

---

#### C2. API NEXUS bind em 0.0.0.0 com rotas admin de controlo de engines
**Ficheiro:** `run_api.py:13`, `api/server.py`, `api/routes.py:393-471`

```python
# run_api.py:13
uvicorn.run(app, host="0.0.0.0", port=8000)
```

O servidor expõe em **todas as interfaces de rede** endpoints como:
- `POST /api/trading/start` — spawn de engine (admin)
- `POST /api/trading/stop` — taskkill de engine (admin)
- `POST /api/auth/register` — qualquer um pode registar utilizadores (role="viewer" por default, mas cria base para escalar se existir bug noutro endpoint)
- WebSocket `/api/live/ws` — streaming de estado

Mitigações presentes:
- JWT auth + bcrypt em `api/auth.py`, com `require_admin` nos endpoints críticos.
- SECRET_KEY ephemeral se `AURUM_JWT_SECRET` não estiver setado ou for o default inseguro (`api/auth.py:15-26`) — bom: não aceita o default "aurum-dev-secret-change-in-production".
- CORS restritivo (default só localhost:5173/3000).
- Rate limit 100 req/min por IP (`api/server.py:52-76`) — in-memory, reset por restart.
- Pré-start risk gate ligado (`api/routes.py:408-420`).

Porém: bind em `0.0.0.0` significa que **qualquer dispositivo na mesma LAN/VPN** (ou qualquer serviço que faça port-forward inadvertido — comum em setups caseiros) vê a superfície de ataque. JWT quebra = start engine live. SECRET_KEY ephemeral também invalida todos os tokens a cada restart (quebra UX) mas é mais seguro.

**Recomendação:** mudar para `host="127.0.0.1"` por default, e expor em LAN só mediante flag explícita (`AURUM_API_BIND=0.0.0.0`). Adicionar log de startup com o bind actual. Considerar firewall rule no Windows que limite a porta 8000 ao loopback.

### High

#### H1. `pickle.load` em ficheiros de cache sem verificação
**Ficheiro:** `core/cache.py:50, 69`

```python
with gzip.open(p, "rb") as f:
    df = pickle.load(f)
```

Qualquer processo capaz de escrever em `data/.cache/*.pkl.gz` executa código arbitrário dentro da engine AURUM no próximo fetch. Em Windows + OneDrive, este directório é escrevível por qualquer app do utilizador (e pelos clientes OneDrive noutros dispositivos se alguém fizer share do directório pai).

**Recomendação:** trocar por parquet (via `pyarrow`) ou `feather`, ou usar `pickle` com HMAC signing do payload. Se pickle for mantido por legacy, documentar explicitamente que `data/.cache/` é uma zona de confiança e não deve sair do próprio host.

---

#### H2. Modo LIVE aceita fallback plaintext com apenas WARNING
**Ficheiro:** `engines/live.py:256-283`

```python
# 1. Tenta encrypted
if encrypted_path.exists() and pw:
    ... unlock + return
# 2. Cai para plaintext e continua
level = logging.WARNING if mode in _REAL_MONEY_MODES else logging.INFO
log.log(level, f"Keys loaded from PLAINTEXT store for mode '{mode}'")
```

`_REAL_MONEY_MODES = {"live", "arbitrage_live"}` (linha 108). Se o encrypted falhar silenciosamente (`AURUM_KEY_PASSWORD` não setado, cryptography não instalado), o engine arranca em LIVE lendo plaintext com apenas um WARNING no log que pode escapar à atenção do operador.

Contraste: `_guard_real_money_gates` (linha 111-120) REFUSES start para modos reais quando `risk_gates.json` está com defaults permissivos. A mesma disciplina deveria aplicar-se ao key store.

**Recomendação:** para LIVE, exigir encrypted store (`raise RuntimeError` quando não há) em vez de warning. Alternativamente, variável explícita `AURUM_ALLOW_PLAINTEXT_LIVE=1` que o operador tem que setar conscientemente.

---

#### H3. `site_runner.py` usa `shell=True` com comando configurável
**Ficheiro:** `core/site_runner.py:170-176`

```python
popen_kwargs = dict(
    cwd=d,
    ...
    shell=True,
)
self.proc = subprocess.Popen(command, **popen_kwargs)
```

O `command` vem de `self.config.get("command")` ou de `_FRAMEWORK_COMMANDS` (que são literais). O config é lido de `SITE_CONFIG_PATH` (um JSON no disco). Se alguém escrever nesse JSON, qualquer coisa pode ser executada via shell do OS do Joao. Combinado com OneDrive sync, torna-se vector acessível a qualquer dispositivo que tenha o OneDrive.

**Recomendação:** trocar `shell=True` por `shell=False` + `shlex.split(command)`, ou manter shell=True mas validar contra allowlist estrita de comandos. Alternativamente, validar que `command` está contido em `_FRAMEWORK_COMMANDS.values()` antes de executar.

---

#### H4. VNC password hardcoded no `docker-compose.yml`
**Ficheiro:** `docker-compose.yml:13`

```yaml
- VNC_PW=aurum2026
```

O serviço MT5 expõe VNC em `3000:3000` e RPyC em `8001:8001`. Password "aurum2026" é brute-force em segundos. O comentário indica que o operador loga o broker MT5 via browser, o que significa que credenciais de broker passam por este canal.

**Recomendação:** substituir por variável de ambiente (`VNC_PW=${AURUM_VNC_PW}`) e não ligar as portas 3000/8001 a `0.0.0.0` — em Docker Compose, bind a `127.0.0.1:3000:3000`.

### Medium

#### M1. API signatures Binance aparecem em URLs (possíveis logs)
**Ficheiro:** `engines/live.py:975`, `core/exchange_api.py:54`

```python
url = f"{base_url}/fapi/v2/balance?{params}&signature={sig}"
r = requests.get(url, headers={"X-MBX-APIKEY": api_key}, timeout=10)
```

A assinatura HMAC SHA-256 é efémera (válida dentro de `recvWindow=5000ms`), mas aparece em URLs que podem ficar em logs de proxies, em `requests` debug logs, ou em capturas de rede. Melhor prática: passar via params argument em vez de concatenar.

**Recomendação:** usar `params={...}` argumento do `requests.get` em vez de construir URL manualmente, e confirmar que nenhum middleware de logging emite `r.url` completa.

---

#### M2. CORS `allow_methods=["*"]` + `allow_headers=["*"]`
**Ficheiro:** `api/server.py:46-47`

Com `allow_credentials=True` e origens limitadas isto é aceitável, mas ainda permite headers custom inesperados. Baixo risco dada a restrição de origens.

**Recomendação:** listar explicitamente os métodos (`GET, POST, OPTIONS`) e headers (`Authorization, Content-Type`).

---

#### M3. Audit trail hash-chain é opt-in e single-writer
**Ficheiro:** `core/audit_trail.py:131-142, 211-223`

`AuditTrail(hash_chain=True)` não é o default. Pior: a cadeia de hash funciona por instância — se duas engines escrevem no mesmo ficheiro mensal concorrentemente, a cadeia quebra (o lock `self._lock` só protege dentro da instância). A engine live usa `hash_chain=True` (linha 872), mas outros engines que importarem `AuditTrail` podem usar `False` por omissão.

Tamper detection: um atacante com write access ao ficheiro pode truncar ou reescrever blocos e, como a chain é append-only sem ancoragem externa (ex: timestamp assinado por 3rd party), o `verify_chain` só detecta edits, não deletes ao final.

**Recomendação:** tornar `hash_chain=True` o default, documentar que cada (engine, month) deve ter single writer, e considerar escrever periodicamente o `last_hash` para um local imutável externo (S3 object lock, ou mesmo commit num repo read-only) como âncora anti-rollback.

---

#### M4. Risk gates podem ser by-passed via ficheiro de config
**Ficheiro:** `core/risk_gates.py:318-341`

`load_gate_config` lê `config/risk_gates.json` e **retorna defaults permissivos silenciosamente** em qualquer erro (OSError, JSONDecodeError, tipo inválido, TypeError). A única defesa é `_guard_real_money_gates` em `engines/live.py:111-120`, que aborta se `cfg.is_default()` em modo real — OK para LIVE.

Porém, um atacante que escreva um JSON válido com thresholds altos (p.ex. `max_daily_dd_pct: 99`) passa pelo guard porque não é `is_default()`. Não há assinatura nem validação semântica ("isto faz sentido para live?").

**Recomendação:** log WARNING explícito se JSON loaded mas com valores >= 90% dos defaults (sugerindo config de teste em ambiente real), e considerar assinatura HMAC do ficheiro com chave derivada do master password.

---

#### M5. Telegram bot sem rate-limit real e sem sanitização de nomes
**Ficheiro:** `bot/telegram.py:322-329, 334-336`

Auth via `chat_id` + `from_id` match (OK), cooldown de 3s entre comandos. Os comandos actuais só lêem estado (não executam trades), então o blast radius é limitado a info disclosure. Se no futuro forem adicionados comandos de controlo (`/stop_engine`, `/flatten`), esta superfície precisa de reforço.

**Recomendação:** documentar na cabeça do ficheiro que comandos destrutivos **não podem** ser adicionados sem 2FA out-of-band; adicionar logging de todos os comandos recebidos para o audit trail.

### Low / Informational

- **L1. `api/routes.py:62-68`** — `_sum_pending` usa f-string para nome de tabela. É seguro porque só é chamado com literais `"deposits"` e `"withdrawals"`, mas vale um comentário "# nosec — hardcoded table names" ou refactor para mapa enum.
- **L2. `engines/live.py:270`** — `json.load` sem validação de schema. Malformação em `keys.json` degrada para `sys.exit(1)` em vez de fallback, o que é aceitável.
- **L3. `core/key_store.py:101-118`** — PBKDF2 com 480k iterações + SHA256. Dentro das recomendações actuais OWASP (600k em 2023, 480k aceitável). Revisitar em 2027.
- **L4. `core/connections.py:108-140`** — cache de ping com TTL de 8s; sem ameaça directa, mas qualquer throttle-side-channel poderia fazer fingerprinting de actividade. Informacional.
- **L5. `launcher.py` tem 12.524 linhas** — não auditado em profundidade. Considerar divisão em módulos para facilitar revisões futuras.
- **L6. `pyproject.toml`** — dependências com ranges largos (`numpy>=1.24,<3`, `requests>=2.28,<3`). Não há lock file (poetry.lock / requirements.txt) visível na raiz. Em Python 3.14 global (sem venv — ver user memory), isto torna rebuilds não-reprodutíveis. Considerar `pip freeze > requirements.lock.txt` e commit desse artefacto.
- **L7. `core/fs.py:robust_rmtree`** — OneDrive-safe deletion. Bom.
- **L8. `engines/live.py:1277-1405`** — kill-switch orchestration bem pensada: flatten FIRST (evita posição exposta sem stop) + cancel de ordens pendentes + `self.running=False` + audit row ANTES das operações destrutivas. Ver também `_startup_reconcile` (linhas 1033-1088) que aborta o engine se broker state diverge de local — belt-and-suspenders.
- **L9. `core/proc.py:_is_alive`** — defesa contra PID recycling no Windows via FILETIME creation_time + image_name. Bem pensado. Incidente histórico (taskkill acidental de processo não relacionado) justifica a paranoia.
- **L10. `config/keys.json` tem permissions check em POSIX** (`engines/live.py:262-266`) mas não em Windows. NTFS ACLs são diferentes — considerar usar `icacls` para validar owner-only no arranque.

## Pontos fortes

- **Secrets reais nunca entraram em git history** (confirmado via `git log -S` com os valores concretos dos tokens).
- **`.gitignore`** cobre `config/keys.json`, `config/vps.json`, `config/vpn.json`, `config/alchemy_params.json`, `.env`, worktrees, e dados runtime — disciplinado.
- **`core/risk_gates.py`** — arquitectura limpa: snapshots imutáveis, dataclasses, soft/hard severity, guard explícito em `_guard_real_money_gates`. Defaults permissivos mas loud warning (`log.warning` once per session).
- **Kill-switch orchestration** (`engines/live.py:1277-1405`) — ordem flatten-then-cancel correcta, audit FIRST, notificação Telegram, idempotente via `_kill_switch_active`.
- **Audit trail** append-only com hash-chain opcional, canonical JSON deterministic encoding. Verificável pós-facto.
- **Reconciliation loop** (`engines/live.py:1090-1229`) compara local state vs broker com threshold + streak, audita cada drift, triggera kill-switch em 3 drifts consecutivos.
- **Canary mode** (`engines/live.py:72-98`) — scaling explícito de capital para primeira sessão live com promoção manual.
- **Process identity verification** (`core/proc.py:_is_alive`) — FILETIME + image_name contra PID recycling.
- **JWT secret** nunca aceita default inseguro; ephemeral fallback em vez disso (`api/auth.py:15-26`).
- **bcrypt** (`passlib`) para passwords, não SHA-256 puro.
- **Single-position gate** (`core/risk_gates.py:247-258`) default 10% — hard-coded sensibly conservative.
- **Startup reconciliation** aborta engine se broker diverge de local — defesa contra ghost positions.
- **Pré-start risk gate** no endpoint `/api/trading/start` (`api/routes.py:408-420`) — já está implementado (`evaluate_start_gates`).
- **Sem `eval`/`exec`/`yaml.load` inseguros** no codebase.
