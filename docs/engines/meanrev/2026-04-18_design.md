# MEANREV — Simplified Mean-Reversion Engine

> Resposta direta ao archive do ornstein v1 (2026-04-18 00:15). Hint do Joao:
> "mean reversion com as mms e rsi da pra botar um preco medio na entrada".
> Este engine implementa exatamente isso — nada mais.

## Tese

Mean reversion via deviation do EMA50, confirmada por RSI extremo, alvo
na média. Sem multi-TF consensus. Sem bateria OU/Hurst/ADF/VR. Uma camada
de discriminação (deviation + RSI) em vez de oito AND'd.

Premissa empírica: em cripto 15m, extremos de deviation acompanhados de
RSI confirmatório são estatisticamente menos prováveis de continuar — não
porque o mercado é MR puro, mas porque a combinação "preço X ATRs longe
da média AND RSI em extremo" é regime-neutro o suficiente pra dar edge
positivo. Hipótese testável, não artigo de fé.

## Contra-modelo (não arquiva isso sem validação negativa)

- Se expectativa < 0 em backtest honesto: mean-rev simples NÃO serve pra
  cripto. Arquiva. Lane de mean-rev no AURUM fica fechada.
- Se expectativa ≈ 0 mas com alto sample: é noise-trader. Arquiva.
- Se expectativa > 0 com N razoável: vale tune rigoroso.

## Parâmetros (OrnsteinParams-style, dataclass)

```
deviation_enter: float = 2.0   # z-score threshold (close - ema50) / atr
rsi_long_max: float = 30.0     # long only if RSI <= this
rsi_short_min: float = 70.0    # short only if RSI >= this
atr_stop_mult: float = 2.0     # stop = entry ± atr_stop_mult * ATR
time_stop_bars: int = 96       # bars to hold (96 @ 15m = 24h)
risk_per_trade: float = 0.01   # 1% account risk per trade
tf_exec: str = "15m"
```

Intencionalmente conservador. Se der edge com esses defaults, ótimo; se
precisar tunar, próximo passo.

## Pipeline

```
fetch(15m) → core.indicators.indicators(df)
           → compute deviation = (close - ema50) / atr
           → scan per bar:
              entry_long  = deviation <= -deviation_enter AND rsi <= rsi_long_max
              entry_short = deviation >= +deviation_enter AND rsi >= rsi_short_min
              target = ema50 at entry bar
              stop   = entry ± atr_stop_mult * atr
           → simulate trade path (time stop at N bars)
           → apply C1+C2 cost model
```

## Testes unitários

- `test_entry_long_signal_fires` — sintético: close 2 ATR abaixo do ema50 + rsi 25 → long
- `test_entry_short_signal_fires` — mirror
- `test_no_signal_in_normal_range` — deviation 0.5 → no trade
- `test_rsi_gate_blocks_long` — deviation extremo mas rsi 50 → no trade
- `test_stop_triggers_loss` — path simulation
- `test_target_triggers_win` — path simulation
- `test_time_stop_closes_open_trade` — no hit → close at bar N
- `test_costs_applied_to_pnl` — PnL considera SLIPPAGE+SPREAD+COMMISSION

## Critério de sucesso (smoke test inicial)

Rodar `BTCUSDT,ETHUSDT,BNBUSDT,SOLUSDT,XRPUSDT / 180d`:

- `N_trades >= 30` (sample viável)
- `Sharpe > 0.5` (> random noise)
- `WinRate >= 40%` (mean-rev tipicamente tem WR 40-60% com RR favorável)
- `Expectancy (R) > 0` (positiva)

Se passa os 4: promissor, abre lane de tune formal (novo spec, grid
fechado, DSR, etc). Se falha em qualquer um: arquiva e fecha lane
mean-rev no AURUM (engine simples já é o baseline mínimo; se nem ele dá
edge, não há edge a capturar).

## Escopo fora

- Multi-TF: não.
- Divergência consensus: não.
- OU/Hurst/ADF/VR: não.
- Regime gating (macro BTC trend): não por enquanto. Pode adicionar depois
  se simple version mostrar sinal promissor em parte do tempo.
- Kill-switch portfolio-level: não (é backtest-only engine por enquanto).
- Sizing convexo: não — size fixo por `risk_per_trade / stop_distance`.

## Diagnóstico

Sinal `(close - EMA50)/ATR em extremo + RSI em extremo` é essencialmente
**ruído com ligeiro viés trend-continuation** em cripto 15m/1h majors
2025-2026. Três evidências convergem:

1. **Mean-rev direction (37% WR):** perde 63% das vezes — mais que
   aleatório.
2. **Reverse direction (52% WR):** ganha 52% das vezes — levemente
   melhor que aleatório, mas não o suficiente pra cobrir 0.1%
   round-trip cost em 1900 trades.
3. **Expectancy por trade próximo de zero em ambos**: -0.12R a -0.17R —
   o sinal filtra ~2% das barras (1700-2000 trades em 86k bars disponíveis)
   mas as 2% filtradas são estatisticamente indistinguíveis de noise.

**Conclusão (consistente com archive do ornstein v1):** o par "deviation
extrema + RSI extremo" em cripto recente não identifica mean-reversal
nem trend-continuation com edge suficiente. **Não é param tuning que vai
salvar** — o problema é o próprio sinal de trigger.

## Decisão

**ARCHIVE o lane mean-rev no AURUM**, com 2 caminhos abertos pro
futuro:

1. **Regime-gated mean-rev** — hipótese nova: o sinal pode ter edge
   APENAS em regime específico (ex: low-vol sideways). Requer novo engine
   com regime detector (Hurst ou HMM-style). Escopo separado.

2. **Alternative anchors** — trocar EMA50 pelo VWAP diário, ou usar
   volume-weighted levels em vez de MM. Hipótese também separada.

Ambos são engines novos, não tune do `meanrev.py`. Escopo futuro quando
Joao tiver tempo e apetite. Por ora: mean-rev simples não entrega.
